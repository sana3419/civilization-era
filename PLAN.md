# 《文明纪元》技术重规划 — 从 UE5 转向 2D 像素方案（v3）

> 基于原 UE5 设计文档（21 个系统不变），本文只规划**技术栈替换**和**受影响的设计调整**。
> 玩法设计（资源链、政策、外交、英雄等数值系统）与引擎无关，原文档继续有效。
> v2 变更：合入独立技术审查的 7 项必改意见（多线程、存档前置、排期修正、压测补全等）。
> v3 变更：GDExtension 三平台编译与 CI 前置到第零阶段、美术降级梯子、确定性回放边界封口、Golden 回归测试、Steamworks ARM 限制等工程周边修订。

## 一、核心技术决策

### 1.1 引擎：Godot 4.6（锁定版本，开发中途不升级）

| 候选 | 结论 | 原因 |
|------|------|------|
| **Godot 4.6** | ✅ 选用 | 官方自 4.3 起提供 Linux ARM64 编辑器（可在 Jetson Orin Nano 上直接开发，JetPack 6 的 Vulkan 1.3 驱动满足要求）；2D 管线成熟；免费开源；GDExtension 可写 C++ 性能核心 |
| Unity | ❌ | Linux 编辑器仅支持 x86_64 Ubuntu，无 ARM64 版，开发机跑不了 |
| UE5 Paper2D | ❌ | 已放弃，且 Linux 编辑器同样仅 x86_64、2D 支持弱、太重 |
| Bevy (Rust) | ❌ | 无编辑器，UI/工具链成本太高 |

注意：Jetson 8GB 为 CPU/GPU 共享内存，"编辑器 + 运行游戏"同开会比较紧，属可用但非舒适。

**语言分工**（对应原方案 "C++ 核心 + 蓝图辅助"）：
- GDScript：UI、事件、任务、外交等低频逻辑
- C++ (GDExtension)：单位模拟、寻路、战斗结算、地图生成（热路径）
- **模拟核心从第一天起就搭 C++ 骨架**。以 GDScript 的速度，模拟核心约 1000 单位即撑不住，"先全 GDScript 后下沉"的路线不成立；GDScript↔GDExtension 边界调用成本在第零阶段实测

### 1.2 渲染：序列帧像素动画（Sprite Sheet）

"6 个像素图动态加载" = **序列帧动画 + 纹理图集 + 按需异步加载**：

- 每个单位每个动作 = 一张精灵图集（Sprite Sheet），如行走 6 帧、攻击 4 帧、死亡 4 帧
- 美术工具：Pixelorama（本身是 Godot 程序，ARM 上能跑）或 LibreSprite；Aseprite 无官方 ARM Linux 二进制，需自行编译，作备选
- 方向处理：俯视角 **4 方向**（左右镜像省一半图）；6 帧 × 4 方向 × 3 动作 ≈ 72 帧/兵种
- 动态加载：`ResourceLoader.load_threaded_request()` 异步加载图集；同 Chunk 单位共享同一份纹理
- **大规模渲染**：同兵种一个 `MultiMeshInstance2D`（单 draw call），开 `use_custom_data` 把帧索引/朝向写入每实例 CUSTOM 数据，canvas_item shader 用 `INSTANCE_CUSTOM` 偏移 UV 采样图集
- **关键前提**：万级实例的每帧更新不能逐个调 `set_instance_transform_2d()`（GDScript 下这一步先于渲染成为瓶颈），必须用 `RenderingServer.multimesh_set_buffer()` 整块上传 PackedFloat32Array，或在 C++ 直填 buffer

### 1.3 模拟架构：逻辑与渲染分离 + 多线程

替代原方案的 Mass Entity：

```
[模拟层] C++ (GDExtension)，独立 worker 线程，固定步长 tick（SIM_TICK_RATE 可配参数，默认 10Hz）
  ├─ 单位数据：紧凑数组（SoA），不挂场景树，从设计之初即受"可序列化"约束
  ├─ 寻路：网格 A*（少量单位）+ 流场 Flow Field（大军团）
  │   流场按 64×64 Chunk 分块缓存，生成跨多 tick 摊销
  ├─ 碰撞：空间哈希网格 + 圆形推挤（不用物理引擎）
  ├─ 战斗/经济结算；万级单位 AI 用模拟层内轻量状态机/效用函数
  └─ 6 核并行：按 Chunk 或系统切分任务（Orin Nano = 6× Cortex-A78AE @1.5GHz，单核弱）
[渲染层] 主线程每帧读模拟快照插值 → MultiMesh buffer 整块上传
[UI 层]  Godot Control 节点，读模拟层快照
```

- **目标地图尺寸：512×512**（流场/迷雾/寻路成本均按此估算；1024×1024 为压测达标后的可选项）
- 模拟层不依赖渲染帧率与系统随机源（统一种子 RNG）——为回放调试留口子
- **确定性边界（开工前封口）**：回放**仅限同构机器调试用途**。ARM 开发机录的回放在 x86 上放会因浮点行为差异（libm 实现、FMA 合并）漂移；玩家侧回放/观战/锁步联机**明确不做**——若未来要做，模拟核心须改定点数或严格浮点纪律（`-ffp-contract=off`、自带三角函数实现），属事后补不了的决定
- tick 频率不写死：10Hz 对 RTS 经济/移动够用；若近战打击感发肉，提到 15-20Hz 成本是线性的，调参数即可
- 原方案的 Chaos 物理、Nanite、Lumen 全部不需要；分块、对象池、分帧 AI 等优化思路原样保留
- **数据驱动**：兵种/建筑/科技/政策全部外置数据表（JSON/Resource），既方便平衡性调整也天然预留 Mod 入口（是否开放 Mod 在可玩切片后决策，事后加远贵于事先留）

## 二、受 2D 化影响的设计调整

| 原设计（UE5 3D） | 2D 像素方案 |
|------------------|-------------|
| 骑砍2 式 3D 物理碰撞战斗 | 俯视角 2D 实时战斗，圆形碰撞 + 推挤；阵型/冲锋/士气机制**完整保留**，冲锋速度加成按 2D 速度算 |
| 城墙有高度、墙上驻军射击 | 城墙为特殊地块，驻军获得"墙上"状态（防御+80%、射程+2）；**反制规则**：攻城器械对墙上单位伤害×1.5，防止攻城退化为纯消耗 |
| 攻城塔/云梯爬墙 | 简化为"贴墙后开辟登墙点"，单位经登墙点进入墙上状态 |
| 相机 WASD+旋转+缩放 | WASD 平移 + 滚轮缩放（像素游戏不做旋转） |
| 地形高度差（丘陵高处攻击加成） | 保留为地块属性加成，视觉用色调/阴影表现 |
| 弹道物理（箭矢/投石） | 2D 抛物线插值表演 + 落点判定，无真物理 |

其余 21 个系统数值与规则不变，仅实现载体从 UE5 类换成 Godot 节点/C++ 模块。

## 三、关键系统的 Godot 落地方式

| 系统 | Godot 实现 |
|------|-----------|
| 地图地形 | `TileMapLayer`（多层：地形/资源/道路/建筑），Perlin 噪声用内置 `FastNoiseLite` |
| 战争迷雾 | 一张低分辨率纹理 + shader，按视野分帧更新 |
| 单位渲染 | MultiMeshInstance2D 按兵种分桶，buffer 整块上传；选中圈/血条用另一层 MultiMesh |
| RTS 框选/编队 | 屏幕矩形 → 空间哈希查询；编队存单位 ID 数组 |
| UI（文明6风格） | Godot Control + Theme；科技树按**自绘**估期（GraphEdit 是编辑器向控件，玩家向 UI 大概率要重做） |
| 寻路 | 自实现网格 A*（地形移动消耗表做权重）；大军团用分块缓存流场 |
| 存档 | 模拟层 SoA 数组直接序列化为二进制（含版本号字段）+ `FileAccess` 内置 Zstandard 压缩；JSON 留给调试 |
| 行为树 AI | LimboAI 插件（**pin 到匹配 Godot 4.6 的 release，与引擎一起锁死**）——**仅用于低频对象**（阵营战略 AI、英雄、任务脚本，数量≤几十个）；万级单位 AI 在 C++ 模拟层用状态机/效用函数，不进场景树 |
| 音频 | 万级单位需发声裁剪：同种音效限流（如同类音效同时≤4 个）、按镜头距离裁剪、总混音预算 |

## 四、美术资产规划

- 风格基准：16×16 或 24×24 像素/单位，地块 32×32（密度参考铁锈战争/环世界；注意铁锈战争大部分单位是**单张图旋转**而非方向序列帧——这是它能单人做出来的原因之一）
- 单位动画规格（满配目标）：待机 2 帧、行走 6 帧、攻击 4 帧、死亡 4 帧，4 方向（左右镜像，实绘 3 方向）
- **美术降级梯子**（与性能降级同等地位，在切片决策点和玩法一起裁）：
  1. 满配：3 绘制方向序列帧 ≈ 16 帧 × 3 方向/兵种，30+ 兵种即上千帧手绘——**单人项目真正的进度杀手**
  2. 降一档：仅侧面 1 方向序列帧 + 左右镜像
  3. 降两档：旋转单图——载具、攻城器械、船只**天然适合**旋转单图，优先降这些；人形单位优先保留序列帧
  - 占位期全部用旋转单图先跑通流程
  - **已决策（2026-06）**：开发期全程使用旋转单图，序列帧补帧推迟到项目完成后再做
- 建筑：静态图 + 1~2 帧工作动画，建造中用半透明蓝图态
- 图集打包：按兵种或建筑类别打包，单图集 ≤2048×2048；图集等二进制资产走 **Git LFS**
- 兵种换色复用：palette swap shader（同骨架不同调色板）
- 起步用免费素材占位：Kenney.nl、itch.io 的 RTS/中世纪像素包
- **中文像素字体**：16px 级别可选项少（方舟像素、缝合像素等），第一阶段搭 UI 框架时与 Godot Theme/CJK 字号一起定，同时建立翻译流程（CSV/PO）——不能事后补

## 五、目标平台与发布

- 玩家平台：**Windows x86_64（Steam）**为主，Linux x86_64 顺带
- **交叉导出"无障碍"只对引擎本体成立**（导出模板是预编译的）。自己的 C++ 模拟核心是 GDExtension 动态库，必须为每个目标平台单独编译：
  - 三个目标：`linux.arm64`（开发机）、`windows.x86_64`（Steam）、`linux.x86_64`
  - Jetson 上交叉编 Windows 版用 **llvm-mingw**（对 ARM64 宿主机支持比 mingw-w64 省事）
  - `.gdextension` 文件按平台填三份库路径
- **CI：GitHub Actions 每次 push 自动编译三平台动态库**，并自动产出 x86_64 整包——"每里程碑导一次包"由 CI 自动化；产出包在一台普通 x86 PC 上冒烟测试，行为与性能须在真实目标平台验证
- Jetson 本地首次全量编译 godot-cpp 不快，配 **ccache**
- **Steamworks SDK 在 ARM Linux 上完全跑不了**：所有 Steam 功能（成就、云存档等）只能在 x86 冒烟机上调试，第五阶段排期单独留出
- 隐藏利好：开发机弱于目标玩家机型，Jetson 上达标 ≈ 玩家机上有富余

## 六、开发阶段

> 排期原则：第零~二阶段为可信估期；**第三~五阶段在"可玩切片"验证后再正式排期**。
> 全部 21 系统做完是 4-5 人年的量，单人路线必须以切片决策点裁剪范围。

### 第零阶段：技术验证（2 周，允许溢到 3 周——闸门的质量比准时重要）⚠️ 硬闸门

**0. 工具链先行**（压测代码本来就是 C++，第一次就配齐）：
- godot-cpp + GDExtension 三平台编译跑通（llvm-mingw 交叉编 Windows）
- GitHub Actions CI：push 即编三平台 + 出 x86_64 包
- ccache、Git LFS

压测项（任一不达标即降目标或改架构）：
1. MultiMesh 渲染 10000 个 6 帧动画精灵 + **每帧 buffer 整块上传**（测更新路径，不只测静态渲染）
2. 流场寻路原型：1000 单位同时移动，512×512 地图
3. GDScript↔GDExtension 边界调用成本基准
4. Forward+（Vulkan）vs Compatibility（GL）渲染器在 Jetson 上对比（纯 2D 后者有时更快）
5. **6 核并行模拟**：C++ worker 线程跑 10000 单位 tick，测主线程交接开销（依赖先搭出 C++ 骨架雏形，排最后）

不达标的降级路径：单位目标 10000 → 3000；地图 512 → 256。

**✅ 闸门结果（2026-06-11，全部通过，无需降级）**：
| 压测项 | 结果 | 达标线 |
|--------|------|--------|
| 10000 动画精灵渲染 + 每帧 buffer 整块上传 | avg 283 FPS / worst 275 | 30 FPS |
| 512×512 全图流场生成 | 17.5 ms/次（按命令触发，非每 tick） | 可摊销 |
| 流场跟随 tick | 1000 单位 0.8ms / 10000 单位 2.2ms | <100ms |
| 模拟 tick（10k 单位，碰撞+移动） | 1 线程 2.2ms / 6 线程 1.7ms | <100ms |
| GDScript↔C++ 边界 | ~48ns/call（万次/帧循环不可行，验证架构） | - |
| 渲染器对比 | Vulkan Mobile 283 > GL Compat 260 → **定 Vulkan Mobile** | - |
| 确定性 | 同种子 + 线程数不变性均 PASS | PASS |

注：每 tick 现场起线程导致多线程加速比差（6 线程仅 1.3×），生产换持久线程池；
绝对值余量巨大，不阻塞。测试环境为无头 X（虚拟 640×480），真实显示器分辨率更高
但 16px 精灵填充率占比极小，结论不变。

## 七、风险清单

1. **Jetson 性能上限**：6 核 A78AE 单核弱、8GB 共享内存，模拟必须 C++ 多线程 + 数据导向；第零阶段压测是硬闸门，降级路径已定义
2. **范围过大**：4-5 人年的量，对策已落实为第二阶段末的可玩切片决策点 + 三~五阶段切片后再排期
3. **美术量是单人项目真正的进度杀手**：满配规格约上千帧手绘；对策 = 第四节的美术降级梯子（3 方向 → 1 方向镜像 → 旋转单图）+ 占位期全旋转单图 + 换色复用，在切片决策点与玩法一起裁
4. **流场寻路 + 阵型移动**的结合是全项目最难技术点，已在第二阶段单列并加权估期
5. **阵营 AI 工作量易被低估**：先做最小可用版本，强 AI 留作切片验证后的扩展项
